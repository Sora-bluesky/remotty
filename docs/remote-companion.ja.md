# Telegram ブリッジとしての方針

`remotty` は、Windows 上の Codex 作業を Telegram から見守るためのブリッジです。
Codex App の代替ではありません。

AI に任せた作業は、ずっと自動で進むとは限りません。
承認が必要になったり、判断待ちになったり、ひとこと補足すれば進められる場面があります。

`remotty` は、その小さな接点だけを Telegram へ持ち出します。
全文の確認、差分の詳細レビュー、プロジェクト選択、細かい操作は、Codex App または `Codex CLI` が担います。
対象は、通知、承認の中継、短い状態確認、短い追加入力です。
今後も、Codex App Server を使う構成に合う形で進めます。
ただし、Codex の作業画面そのものを置き換えることは目指しません。

## プロダクトの位置づけ

Codex App または `Codex CLI` は、主な作業画面です。
`remotty` は、外出先から短く声をかけるための Telegram ブリッジです。

`remotty` は、次の問いへすばやく答えられる状態を目指します。

- Codex はまだ動いているか
- 今何をしているか
- 承認が必要か
- 止めるべきか、続けるべきか
- 一言だけ追加で伝えるべきか
- 作業は完了したか、失敗したか

## 現在の範囲

現在の公開手順では、Windows 上のローカル `Codex CLI` セッションと Telegram をつなぎます。
推奨する接続方式は次の設定です。

```toml
[codex]
transport = "app_server"
```

この設定では、`remotty` がローカルの `codex` コマンドを使います。
また、`app_server` 接続を通じて、スレッドの継続、承認の中継、追加入力を扱います。

起動済みの Codex App Server へ接続する機能は、今後の方針です。
現在のクイックスタートでは必須ではありません。

`exec` 接続は、高度な予備手段として残します。

## 目指さないこと

`remotty` は、次のものを目指しません。

- 全文を読む UI
- 詳細な差分ビューア
- エディタ統合
- ファイルブラウザ
- ワークツリー管理
- プロジェクト管理 UI
- Codex App の複製

これらは、より高機能な Codex の画面が担う領域です。

## 今後の進め方

今後の実装でも、同じ境界を守ります。

- 利用可能なら、既存の Codex App Server へ接続する
- Windows サービスとして常駐する時は、ローカルの `app_server` を起動する
- Telegram のチャットと Codex のスレッドを紐づける
- 更新頻度を抑えたライブカードを維持する
- 許可済み送信者だけが承認を操作できるようにする
- 実行中のターンへ短い追加入力を送る
- Telegram が通知で埋まらないよう、差分とイベントを要約する
- `app_server` の API が安全に使える場合だけ、`goal` 状態を表示する

App Server の API が不明確な場合や試験的な場合は、安全に使える明示的な API だけを使います。
挙動を推測して、未知のスラッシュコマンドをスレッドへ注入しません。

## 安全性の境界

Telegram の認可は、送信者単位で維持します。
特にグループチャットでは、`chat_id` だけでは不十分です。

既定では、次を守ります。

- bot token は Windows の保護領域へ保存する
- ペアリング済みで、allowlist に入った送信者だけが操作できる
- 承認操作は、許可済み送信者だけが実行できる
- 実行時の状態は `%APPDATA%\remotty` 配下へ保存する
- プロジェクトファイルと `remotty` の状態ファイルを分ける

## 通知方針

新しい Telegram メッセージは、重要なイベントに限ります。

- 承認が必要になった
- 作業が完了した
- ターンが失敗した
- `app_server` 接続が切れた
- 長時間止まっている
- ユーザー入力が必要になった

通常の進捗、出力の差分、繰り返しの状態更新は、編集されるライブカードか短いイベントログへまとめます。
